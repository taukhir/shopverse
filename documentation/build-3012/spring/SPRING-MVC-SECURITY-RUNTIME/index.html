Spring MVC And Security Request Runtime

A servlet request enters container filters before DispatcherServlet. Spring Security's FilterChainProxy selects one SecurityFilterChain; ordering and request matchers decide which policy applies. Authentication filters build a token, delegate to AuthenticationManager/providers, and place a successful authentication in SecurityContext. Authorization occurs later at URL or method boundaries.

DispatcherServlet selects a handler through mappings, obtains an adapter, invokes argument resolvers, validation and controller logic, then processes return values through handlers/message converters. Content negotiation selects representation; Jackson serialization happens after controller return and can trigger lazy-loading or recursion if entities escape.

Filters wrap the entire servlet chain; interceptors surround handler execution; controller advice handles mapped application exceptions; servlet error dispatch can occur when failure escapes. Authentication failures are commonly translated inside security exception handling before MVC advice.

Async MVC releases the container request thread while work continues, but context, timeouts, cancellation and response completion need explicit ownership. An executor queue can become the hidden latency source. Virtual threads simplify blocking execution but do not replace connection/rate limits.

JWT resource-server validation must verify signature, issuer, audience where required, time claims and algorithm policy. Do not treat decoded claims as trusted before validation. Method security is another proxy/interceptor boundary and can be bypassed by self-invocation in the same ways as other proxy advice.

Tricky Interview Questions​

  1. Filter versus interceptor? Filters are servlet-chain infrastructure; interceptors are handler-aware MVC infrastructure.
  2. Why might @ControllerAdvice not handle an auth failure? Security translated it in the filter chain.
  3. Can two security chains apply? The first matching chain is selected.
  4. Why can returning an entity cause SQL after the service method? Serialization touches lazy state.
